-Diseño e ingeniería modulo de Incidentes
-
-Reglas Generales
-
-Utiliza el layout para el registro de incidentes tipo acordeon, la primera vez que se carga y se graba se llama semilla Original. Cuando se graba y se cierra y luego se vuelve a abrir se llama Semilla Base.
-
-Este es uno de los modulos mas problemáticos, por tanto debe estar contenido en un .py separado.  Para funcionar se deben seguir las siguientes reglas :
-
-a.- Cuando se carga por primera vez se genera un índice único, que acompañara toda la vida del incidente, este índice debe ser con la nomenclatura
-
-String :
-
-NRO CORRELATIVO INCREMENTAL
-RUT EMPRESA SIN DIGITO VERIFICADOR
-NUMERO DEL MODULO
-NUMERO DEL SUB MODULO
-DESCRIPCION DEL CUADRO DE DIALOGO O RUTA Y NOMBRE DEL ARCHIVO
-
-- Este índice se debe grabar cada vez que se crea un incidente
-- Este indice se utiliza de referencia cada vez que se edita un incidente
-- Todos los datos se mantienen en un archivo temporal mientras se edita el incidente, sino se graba o se sale por accidente o se interrumpe el programa, se trabaja con el indice original
-- Los archivos y el indice se guardan en carpetas upload separadas segun normas de ciberseguridad
-- El indice y demas datos excepto archivos se guardan en la base de datos
-- El sistema es multitenan
-- Cuando se genere la edicion se debe considerar este indice como unico referencia de carga, al editar el despliegue de este incidente se debe construir a partir del indice
-

-4.1.- Selección de taxonomía (viene de la tabla taxonomia_incidentes) , se despliegan dependiendo del tipo de empresa si es OIV o PSE, para este criterio, la tabla contiene como tipo OIV, PSE o AMBAS.

-Del punto 4.2 a 4.4 se puede repetir tantas veces como taxonomías seleccione el usuario, cada una de ellas es una entidad individual.
-
-como son varias taxonomías, podemos tener por ejemplo :
-
-4.2.1 por taxonomía numero 1 de la selección
-4.2.2 por texto de porque fue seleccionada esta taxonomia
-4.2.3 por observaciones adicionales
-4.2.4 por las evidencias que contiene la referencia del archivos, comentarios y fecha. Importante, este concepto se convierte en 4.2.4.1 para el archivo uno, 4.2.4.2 archivo 2 y asi sucesivamente según la cantidad de archivos que se suban
